# Decision Log

> normalized-growth-skill에서 정공 플레이북을 우회하거나, 사용자 결정이 필요한 분기점에서의 결정을 기록.

## D-001 — Existing + Growth, audit 게이트 우회

- **날짜**: 2026-04-20
- **상황**: clib 앱은 iOS 스토어 출시 직후(growth). normalized-growth-skill의 엄격 모드는 existing 프로젝트가 growth로 진입하기 전 50_existing_audit(10스텝)을 요구.
- **결정**: 이번 세션은 audit 풀 플레이북을 **우회**하고, 사용자의 단일 명시 요구(랜딩 페이지 리뉴얼)만 처리.
- **근거**:
  1. 사용자 요청이 매우 구체적이고 단일 산출물 (앱 홈페이지 + 개인정보처리방침 + 다운로드 버튼)
  2. 기존 베이스라인(`clib-support/docs/index.html`)이 이미 있고, 카피/번역/컬러는 검증된 상태
  3. 풀 audit는 2~3 세션 소요 — 사용자의 즉시 가치를 막음
- **상쇄 조치**: 다음 세션에서 **50_existing_audit Step 1(인벤토리)** 만이라도 수행 권장. PROJECT_STATE에 백로그로 기록.
- **상태**: 사용자 명시 승인 (2026-04-20 메시지 "응 추천한대로 해")

## D-002 — 랜딩 도메인 (대기)

- **날짜**: 2026-04-20
- **상황**: 현재 랜딩은 `https://vljh246v.github.io/clib-support/`. 별도 도메인(예: `clib.app`) 사용 의사 미확인.
- **결정**: **현 GitHub Pages URL 유지**를 기본값으로. 신규 도메인은 향후 결정.
- **근거**: 프로젝트 인프라 재배치 없이 구현 가능. 도메인 변경은 마케팅·SEO 결정으로 분리.
- **상태**: 사용자 확인 필요 시 재논의 (현재는 묵시적 동의로 가정)

## D-003 — Android 미출시 처리

- **날짜**: 2026-04-20
- **상황**: clib은 iOS App Store만 출시. Google Play 미출시.
- **결정**: 다운로드 버튼은 **App Store 단 1개만 표시**. 그 아래 작은 텍스트로 "Android는 곧 출시 예정" 1줄 캡션.

  - Play Store 사전등록 → 사전등록 페이지 미생성 상태
- **상태**: 사용자 추천 채택 (2026-04-20)

## D-004 — Privacy 페이지 분리 여부 (대기)

- **날짜**: 2026-04-20
- **상황**: 기존 페이지는 단일 페이지에 앱 소개 + 개인정보처리방침 통합. Apple-style 풀-블리드 랜딩으로 가면 privacy를 별도 영역(앵커 또는 별도 파일)으로 분리하는 게 자연스러움.
- **선택지**:
  - A) `index.html` 내 `#privacy` 앵커 섹션으로 유지 (단일 파일, 스토어 신청 시 URL 동일하게 유지)
  - B) `privacy.html` 별도 파일로 분리 (랜딩이 더 가벼워짐, 단 App Store Connect의 Privacy URL 갱신 필요)
- **결정**: **A안 채택** (사용자 별도 의견 없을 시 기본값). App Store Connect의 Privacy Policy URL 변경 비용을 피함.
- **상태**: 사용자 확인 필요
